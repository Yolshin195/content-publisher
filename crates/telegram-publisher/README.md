# Telegram Publisher

Микросервис публикации постов в Telegram-канал/группу через бота-администратора.
Реализует контракт `POST /publish` из `shared-contracts` — тот же самый, который
будет реализовывать любой будущий Publisher (VK, X и т.д.), поэтому Core Service
обращается к нему без каких-либо изменений на своей стороне.

Отправка сообщений выполняется через [`teloxide`](https://docs.rs/teloxide) —
используется только клиент запросов (`Bot` + `Requester`), без диспетчинга входящих
апдейтов, т.к. сервису нужно лишь публиковать посты, а не обрабатывать сообщения от
пользователей.

## Запуск

1. Создать бота через [@BotFather](https://t.me/BotFather), получить токен.
2. Добавить бота **администратором** в целевой канал/группу (иначе `sendMessage`/
   `sendPhoto` вернёт `403 Forbidden`).
3. Скопировать `.env.example` в `.env`, вписать `TELEGRAM_BOT_TOKEN`.
4. Запустить:

   ```bash
   cargo run --bin telegram-publisher
   ```

5. Указать адрес сервиса в Core Service через `PUBLISHER_TELEGRAM_URL` (по
   умолчанию `http://localhost:8081`, см. `.env.example` в `core-service`).

## Логика публикации (`src/telegram.rs`)

- `target.external_id` принимает либо публичный юзернейм (`@channel`), либо
  числовой chat_id супергруппы/канала (`-1001234567890`).
- `target.config.parse_mode` — `html` (по умолчанию), `markdown` или `markdownv2`.
- Если у статьи есть `cover_image_url` — публикуется через `sendPhoto` с подписью
  (лимит Telegram — 1024 символа, длинные подписи обрезаются с `…`), иначе —
  обычным `sendMessage` (лимит 4096 символов).
- Каждая ссылка на видео (`video_links`) становится отдельной inline-кнопкой под
  постом с подписью по платформе («Смотреть на YouTube» / «Смотреть в VK Video» и
  т.д.) — это и есть поддержка нескольких зеркал одного ролика на случай, если
  YouTube недоступен у части аудитории.
- `PublishResult.permalink` строится «по возможности»: для публичных каналов —
  рабочая ссылка `t.me/username/id`, для приватных — внутренняя `t.me/c/.../id`
  (открывается только у тех, кто уже состоит в канале).

**Допущение:** `content.body_html` должен быть HTML, совместимым с [ограниченным
набором тегов Telegram](https://core.telegram.org/bots/api#html-style) (`b`, `i`,
`u`, `s`, `a`, `code`, `pre` и несколько других). Полная санитизация произвольного
HTML из редактора статей в этот сервис не входит — это можно добавить позже как
отдельный шаг перед отправкой, если content_html будет содержать более широкий набор тегов.

## Компиляция

Как и для `core-service`, в среде, где готовился код, не было доступа к
Rust-тулчейну (сетевые ограничения песочницы), поэтому `cargo build`/`cargo check`
не запускались. Больше всего стоит перепроверить `src/telegram.rs` — это
единственное место, использующее внешнюю библиотеку `teloxide`, чьи точные пути
типов (`teloxide::types::*`, трейты `...Setters`) я не мог сверить компилятором.
Если `cargo check` укажет на неверный путь импорта — пришлите вывод, поправим.
